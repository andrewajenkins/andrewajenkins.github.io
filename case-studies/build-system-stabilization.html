Systematic Fixes

The Challenge

E2E and unit test pipelines suffered from ~40-60% build success rates, severely impacting developer productivity across the organization.

Root Causes Identified:

  • AWS spot instances terminating unpredictably: Could fail at any stage of multi-stage pipeline, making it virtually impossible to failure-proof the entire build process
  • Git checkout failures: BitBucket API rate limits, EOF errors, stream corruption when multiple child jobs checked out code simultaneously
  • Docker image rebuild waste: Pre-built images weren't being reused, adding ~8 minutes per test suite

Worked systematically over 6-7 weeks to diagnose and fix each failure mode, ultimately improving reliability from 40-60% to 95%+.

The Approach: Layered Strategy

Executed a multi-phase stabilization strategy over 6-7 weeks, prioritizing quick wins first while designing long-term architectural improvements.

Phase 1: Hardware Tier Separation

Separated E2E infrastructure into dedicated hardware tiers. Build/setup stages run on stable hardware; only test execution runs on pullable instances. This prevented spot instance terminations from killing entire multi-stage builds.

Impact: Prevented spot instance pulls from destroying complete build workflows

Phase 2: Git Checkout Resilience

Added comprehensive retry logic and error handling for git operations. Handled EOF exceptions, stream corruption, and implemented fallback mechanisms to survive transient failures.

Impact: Reduced checkout failures by ~70%

Phase 3: On-Demand Migration

Eliminated spot instances entirely, migrating to on-demand infrastructure after leadership prioritized reliability over cost. This quick win immediately stabilized builds while longer-term solutions were designed.

Impact: Eliminated 40-50% of build failures from spot instance terminations

Phase 4: S3 Source Caching Architecture

Designed and implemented S3-based source code caching layer (collaborative effort with infrastructure team). Parent job checks out code once and syncs to S3; child jobs copy from S3 cache instead of repeated git operations, eliminating BitBucket API rate limit issues.

Impact: Reduced git operations by ~90%, near-complete elimination of checkout failures

The Solution

Layered Fixes: Quick Wins + Strategic Architecture

Spot Instance Elimination

Migrated entirely to on-demand instances. Since spot instances could terminate at any pipeline stage, attempting to failure-proof every stage was impractical—elimination was the pragmatic solution.

S3 Source Caching

Solved BitBucket API rate limit problem by caching source code in S3. Parent job checks out once; child jobs copy from S3 cache instead of repeated git operations hitting API limits.

Git Checkout Retry Logic

Added comprehensive error handling for EOF exceptions and stream corruption, with intelligent retry mechanisms to survive transient git failures.

Docker Image Reuse

Fixed agent configuration so pre-built Docker images are properly reused across test suites, eliminating unnecessary ~8 minute rebuilds per suite.

The Impact

Measurable Results

  • Build success rate: 40-60% → 95%+
  • Spot instance failures: 100% eliminated
  • Git checkout failures: ~95% reduced
  • Git operations: ~90% reduced via S3 caching
  • Image rebuild time: 8 minutes/suite saved

Strategic Outcomes

  • Dedicated DevOps team formed as direct result of this work

Lessons Learned

Pragmatism Over Perfectionism

Attempting to failure-proof every stage of a multi-stage pipeline against spot instance terminations was impractical. Sometimes the right solution is elimination, not mitigation.

Layered Strategy Wins

Quick wins (on-demand migration) bought time to design strategic solutions (S3 caching). Don't let perfect be the enemy of good when systems are failing.

API Rate Limits Are Real

Multiple child jobs hitting BitBucket simultaneously caused rate limit failures. Caching at the parent level eliminated redundant API calls and solved the root cause.

Systematic Beats Heroic

6-7 weeks of methodical fixes (hardware tiers → retry logic → migration → caching) proved more effective than rushed, one-off solutions. Infrastructure work requires patience.
